# antigravity: Git Commit & PR 규칙

## 절대 규칙 (Tone / Output)

- 화자는 정직하고 과묵하다.
- 출력은 설명적이지 않고 간결하다.
- 출력은 항상 동일한 구조만 사용한다. 변형 금지.
- 출력은 항상 코드블럭으로 감싼다.
- 커밋 여부를 Y/N로 묻는다. Y면 커밋+푸시까지 실행한다.
- 커밋 전 반드시 변경 파일 목록을 확인한다:
  - `git diff --name-only --cached` 또는 `git status --porcelain`
- 린트 정책:
  - **수정된 파일**의 린트 에러만 무시 가능
  - **수정되지 않은 파일**의 린트 에러는 커밋 차단
- 푸시 완료 후 PR 생성 여부를 Y/N로 묻는다.
  - Y면 `gh pr create` 실행
  - gh 인증이 안 되어 있으면 브라우저로 PR 생성 페이지를 연다

---

## 커밋 메시지 Prefix 규칙 (필수)

Message 앞에 Prefix를 추가한다:

- **[FIX]** 올바르지 않은 동작을 고친 경우
- **[ADD]** 테스트 코드/리소스/비즈니스 로직이 아닌 보조 요소 추가
- **[REMOVE]** 코드/리소스 삭제
- **[RENAME]** 이름 변경
- **[UPDATE]** 버전 업데이트 등 “잘못된 것을 바로잡는 것”이 아닌 업데이트
- **[REFACTOR]** 리팩토링/코드 수정
- **[FEAT]** 기능 추가

---

## 작성 규칙

- “작성/추가/삭제” 같은 행위 단어는 가급적 쓰지 않는다.
- 예시:
  - `[FIX] 일부 개인정보가 파기되지 않는 이슈`
  - `[REMOVE] 개인정보 파기 기능 불필요 코드`
- 세부 설명이 필요하면 **한 줄 공백 후 하이픈 목록**으로 작성한다.
- 한글 인코딩이 적용되지 않으면 영어로 작성한다.
